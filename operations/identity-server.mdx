The Identity Server (IS) is the authoritative registry for every entity in The Things Stack: users, organizations, applications, gateways, and OAuth clients. It exposes the Registry and Access gRPC services, backs them with a PostgreSQL database, and uses an optional Redis cache for high-throughput rights lookups. Every API call made to any other component is ultimately authorized by rights that the IS has granted.

## Entity model

The IS stores five top-level entity types:

| Entity | Identifier field | Description |
|---|---|---|
| User | `user_id` | Human operator or service account |
| Organization | `organization_id` | Group of users; can own applications and gateways |
| Application | `application_id` | Container for end devices and integrations |
| Gateway | `gateway_id` | LoRaWAN packet forwarder |
| OAuth Client | `client_id` | Third-party application using delegated access |

Each entity can have **collaborators** (users or organizations) with a set of rights, and can have **API keys** with a subset of those rights.

## Rights system

Rights are fine-grained string constants prefixed with `RIGHT_`. They control every operation across all entities. A few important examples:

<AccordionGroup>
  <Accordion title="Application rights">
    | Right | Description |
    |---|---|
    | `RIGHT_APPLICATION_INFO` | Read basic application info |
    | `RIGHT_APPLICATION_SETTINGS_BASIC` | Update name, description, and attributes |
    | `RIGHT_APPLICATION_SETTINGS_API_KEYS` | Create and delete API keys for the application |
    | `RIGHT_APPLICATION_SETTINGS_COLLABORATORS` | Manage collaborators |
    | `RIGHT_APPLICATION_DEVICES_READ` | List and get end devices |
    | `RIGHT_APPLICATION_DEVICES_WRITE` | Create, update, and delete end devices |
    | `RIGHT_APPLICATION_TRAFFIC_READ` | Subscribe to uplink and event streams |
    | `RIGHT_APPLICATION_TRAFFIC_DOWN_WRITE` | Schedule downlinks |
    | `RIGHT_APPLICATION_DELETE` | Delete the application |
  </Accordion>
  <Accordion title="Gateway rights">
    | Right | Description |
    |---|---|
    | `RIGHT_GATEWAY_INFO` | Read basic gateway info |
    | `RIGHT_GATEWAY_SETTINGS_BASIC` | Update name, description, and attributes |
    | `RIGHT_GATEWAY_SETTINGS_API_KEYS` | Create and delete API keys for the gateway |
    | `RIGHT_GATEWAY_SETTINGS_COLLABORATORS` | Manage collaborators |
    | `RIGHT_GATEWAY_TRAFFIC_READ` | Read gateway traffic and status |
    | `RIGHT_GATEWAY_LINK` | Connect and forward packets (used by the gateway itself) |
    | `RIGHT_GATEWAY_DELETE` | Delete the gateway |
  </Accordion>
  <Accordion title="User and organization rights">
    | Right | Description |
    |---|---|
    | `RIGHT_USER_INFO` | Read user profile |
    | `RIGHT_USER_SETTINGS_BASIC` | Update name and email |
    | `RIGHT_USER_SETTINGS_API_KEYS` | Manage API keys on the user account |
    | `RIGHT_USER_APPLICATIONS_LIST` | List applications owned by the user |
    | `RIGHT_USER_GATEWAYS_LIST` | List gateways owned by the user |
    | `RIGHT_USER_ORGANIZATIONS_LIST` | List organizations the user belongs to |
    | `RIGHT_ORGANIZATION_INFO` | Read organization profile |
    | `RIGHT_ORGANIZATION_SETTINGS_MEMBERS` | Manage organization members |
    | `RIGHT_ORGANIZATION_APPLICATIONS_CREATE` | Create applications under the organization |
    | `RIGHT_ORGANIZATION_GATEWAYS_CREATE` | Create gateways under the organization |
  </Accordion>
</AccordionGroup>

## Database management

The IS uses PostgreSQL. The schema is managed through a migration system backed by `bun/migrate`.

<Steps>
  <Step title="Run migrations">
    Always run migrations before starting the IS for the first time or after upgrading:

    ```bash
    ttn-lw-stack is-db migrate
    ```

    The command connects to the URI configured in `--is.database-uri`, applies all pending migrations, and prints a status summary.
  </Step>
  <Step title="Check migration status">
    To see which migrations have been applied and which are pending:

    ```bash
    ttn-lw-stack is-db status
    ```
  </Step>
  <Step title="Roll back a migration">
    To roll back the most recently applied migration group:

    ```bash
    ttn-lw-stack is-db migrate --rollback
    ```

    <Warning>
      Rollbacks are destructive. Back up the database before rolling back in production.
    </Warning>
  </Step>
  <Step title="Clean up expired entities">
    Soft-deleted entities are retained for the restore period configured in `--is.delete.restore`. To permanently purge all entities past their retention period:

    ```bash
    # Dry run first to review what will be deleted
    ttn-lw-stack is-db cleanup --dry-run

    # Perform the actual cleanup
    ttn-lw-stack is-db cleanup
    ```
  </Step>
</Steps>

## Creating the first admin user

A new IS database has no users. Create the initial administrator before starting the stack:

```bash
ttn-lw-stack is-db create-admin-user \
  --id admin \
  --email admin@example.com
```

You will be prompted for a password. The account is created in the `STATE_APPROVED` state with full admin rights.

## Managing API keys with the CLI

API keys are the primary way non-human services authenticate to the stack. Each key is associated with an entity and carries a subset of that entity's rights.

<CodeGroup>

```bash Application API key
# Create an API key for reading traffic from an application
ttn-lw-cli applications api-keys create my-app \
  --name "My Data Consumer" \
  --right RIGHT_APPLICATION_TRAFFIC_READ \
  --right RIGHT_APPLICATION_DEVICES_READ
```

```bash Gateway API key
# Create a key for a gateway to connect
ttn-lw-cli gateways api-keys create my-gateway \
  --name "Gateway Link Key" \
  --right RIGHT_GATEWAY_LINK
```

```bash User API key
# Create a personal API key with application creation rights
ttn-lw-cli users api-keys create \
  --user-id admin \
  --name "CI Pipeline" \
  --right RIGHT_USER_APPLICATIONS_CREATE \
  --right RIGHT_USER_GATEWAYS_CREATE
```

</CodeGroup>

<Note>
  The full API key value (prefixed `NNSXS.`) is returned only at creation time. Store it in a secrets manager immediately—it cannot be retrieved later.
</Note>

To list and delete existing keys:

```bash
# List API keys for an application
ttn-lw-cli applications api-keys list my-app

# Delete an API key by its ID
ttn-lw-cli applications api-keys delete my-app <key-id>
```

## OAuth clients

OAuth clients allow users to grant delegated access to third-party applications via the standard OAuth 2.0 authorization code flow. The CLI handles this internally using the built-in `cli` client ID, but you can register custom clients for your own integrations.

```bash
# Create an OAuth client
ttn-lw-cli clients create my-integration \
  --user-id admin \
  --name "My Integration" \
  --description "Reads application traffic for analysis" \
  --redirect-uri "https://myapp.example.com/callback" \
  --right RIGHT_APPLICATION_TRAFFIC_READ

# Approve a client (admin only, when approval is required)
ttn-lw-cli clients set my-integration --state STATE_APPROVED
```

## Organizations

Organizations are a way to share ownership of applications and gateways across a team without giving every member admin rights on those entities directly.

```bash
# Create an organization
ttn-lw-cli organizations create my-org \
  --user-id admin \
  --name "Operations Team"

# Add a member with specific rights
ttn-lw-cli organizations members set my-org \
  --user-id janedoe \
  --right RIGHT_ORGANIZATION_INFO \
  --right RIGHT_ORGANIZATION_APPLICATIONS_CREATE

# List members
ttn-lw-cli organizations members list my-org

# Create an application owned by the organization
ttn-lw-cli applications create org-app \
  --organization-id my-org \
  --name "Organization App"
```
